We understand that there’s a balance between getting things done and getting things right. Our focus is on quickly delivering reliable software. That means we commit to building features and infrastructure we need, but we try to avoid building things we only might need, and we don’t chase huge refactors that don't meaningfully improve our software.

And this might sound obvious, but we think about things before building them. Most of our best ideas have come from a few people scribbling on a whiteboard, not one person staring bleary-eyed at Vim.

We believe code is written for humans, not machines. In addition to being correct, code should be expressive and easy to reason about. At Spring, every line of code is reviewed by a human before it’s committed. Code review isn’t just quality control, it's an opportunity to discuss ideas, provide feedback, and learn from the code of others. We always want to be better engineers.

We know code can be beautiful, but that a beautiful product is more important than a beautiful codebase. In the end, the user experience is all that matters.

Getting Hired

Everyone gets treated with respect.

We try to determine whether Spring is right for the engineer, not just whether the engineer is right for Spring.

We aren't sadists. An interview shouldn't be an interrogation.

We don't hire unicorns, rockstars, ninjas, 10xers, etc. We hire strong engineers that we look forward to working with every day.
